Concepts

Key Concepts

PCOs represent additional work beyond the original contract scope. They fall into two categories and feed directly into WIP calculations for real-time job financial health.

PCO — Potential Change Order

Parent concept

Additional work identified beyond the original contract scope. Begins as a field capture and advances through approval to become an official change order or be invoiced directly.

T&M — Time & Material

PCO Type: T_AND_M

Work already completed, billed on actual time and materials consumed. Represents recoverable revenue that needs to be invoiced to the customer. Replaces the paper T&M sheet.

Pending Quote

PCO Type: PENDING_QUOTE

Proposed future work not yet approved or performed. Represents potential revenue pending customer sign-off. May convert to a formal Change Order when approved.

WIP — Work in Progress

Real-time financial metric

The remaining unbilled contract value, including all approved PCOs. A key metric for understanding job financial health and cash flow.

WIP = Original Contract + Approved COs + Approved PCOs − Billed to Date
Dev: PCOs exist upstream of EstimateChangeOrderRecord. When a Pending Quote PCO is approved, it can optionally be converted to a formal Change Order via the existing CO workflow. T&M PCOs flow directly to invoicing.

PCO Status Workflow — State Machine

Every PCO follows this lifecycle. Status transitions are permission-gated, require specific field validations, and trigger notifications at each step.

Auto-Numbering

Pattern: PCO-{NNN} per job
Examples: PCO-001, PCO-002, PCO-003
Generated on first submit (not on draft creation)
Uses MAX(pcoNumber) + 1 WHERE jobId = currentJob

Scope Rules

OWN Scope

createdById = currentUser.id
Field workers can only see and edit their own draft PCOs

PROJECT Scope

PCO.Job.projectId IN user.projectIds
Supervisors see PCOs on jobs within their assigned projects

ALL Scope

No filter applied
PM, Finance, and Admin see all PCOs across all jobs
Dev: Add to PermissionMap in apps/api/src/models/Permission.ts. Scope field: getScopeTargetField() = "createdById" for OWN scope. Same pattern as RFI/EstimateChangeOrder permissions. Field worker edit restriction (draft only) enforced in model layer transition logic.
