"Total In-Progress Jobs Value" on the main dashboard includes archived jobs, inflating the displayed value. The raw SQL query filters by workflow state but does not exclude archived records.

⚠ Bug: The "Total In-Progress Value" metric ($4.2M) is inflated because the query includes archived jobs. The correct value after fix is approximately $3.6M.

Root Cause — SQL Query

Before (Buggy Query)

SELECT SUM(jlf.jobValue) AS totalValue FROM JobListFeed jlf INNER JOIN Job j ON j.id = jlf.id WHERE j.workflowState = 'inprogress' AND j.isDeleted = 0 -- ⚠ Missing: AND j.isArchived = 0

After (Fixed Query)

SELECT SUM(jlf.jobValue) AS totalValue FROM JobListFeed jlf INNER JOIN Job j ON j.id = jlf.id WHERE j.workflowState = 'inprogress' AND j.isDeleted = 0 AND j.isArchived = 0

Step 1

Add ownership guard (NotificationStatus.ts)

In the editNotificationStatus mutation resolver (line ~48), before calling editItem, use the NotificationStatus model class to fetch the record and verify userId === context.userId().

Architecture rule: All data access in GraphQL resolvers MUST go through the model layer (apps/api/src/models/), never directly through prisma.*. 
editNotificationStatus: async (_, args, context) => {
    const model = new NotificationStatus();
    const status = await model.getItem(args.id);
    if (!status || status.userId !== context.userId()) {
        throw new Error("Unauthorized: Cannot modify another user's notification status");
    }
    return model.editItem(args.id, args.data, context.userId());
}
Step 2

Mark notification on leave request view (LeaveRequests.ts)

In the query resolver that returns a single leave request, add a call to cascadeSetAsRead:

await new Notification().cascadeSetAsRead(
    "LeaveRequest",
    leaveRequestId,
    context.userId()
);

The cascadeSetAsRead method (in apps/api/src/models/Notification.ts line ~439) already finds all Notification records matching referenceModel and referenceId, and only updates statuses where status.userId === currentUserId.

Implementation

Fix the SQL WHERE clause

The getTotalValueOfAllJobsInProgress() runs a raw SQL query against JobListFeed. The current WHERE clause filters by pv.workflowState = 'inprogress' but does not filter out archived jobs.

Fix: add AND j.isArchived = 0 to the WHERE clause, matching the pattern used by getActiveJobsCount() (lines ~214-224) which already has this filter.

Note: The JobListFeed view itself (apps/api/prisma/views/JobListFeed.sql) filters j.isDeleted = 0 and p.internalProject = 0 but intentionally does NOT filter j.isArchived. The filter must be applied at the query level.
