Interface 2

Job Schedule — Current vs. Enhanced Filters

Route: /scheduling — The Job Schedule currently applies User Group and Project Lead filters client-side, loading ALL data first. Moving to server-side filtering dramatically reduces payload and improves load time.

⚡
Performance Impact: Client-side filtering loads ALL schedule data (~2-4 MB for 75+ users), then filters in JavaScript. Server-side filtering queries only matching records (~200-400 KB). Expected improvement: 3-5× faster load on filtered views.

Security

SQL Injection Prevention

Both filters involve dynamic SQL construction. All values MUST be parameterized using Prisma's tagged template system. This is non-negotiable.

🛡 Mandatory SQL Security Rules

✅
ALWAYS use Prisma.sql tagged templates for all dynamic SQL values. This ensures automatic parameterization and escaping.
✅ Good: Prisma.sql`WHERE userId = ${userId}`
✅
ALWAYS use Prisma.join() for array parameters (e.g., multi-select filter values).
✅ Good: Prisma.sql`WHERE id IN (${Prisma.join(groupIds)})`
❌
NEVER use string interpolation for SQL values. No template literals, no .concat(), no manual escaping.
❌ Bad: Prisma.sql`WHERE userId = '${userId}'`
❌ Bad: `WHERE userId = '` + userId + `'`
📋
Reference the existing buildJobProjectExistsClause pattern at line ~885 in timesheetMatrixView.ts. This function demonstrates the correct approach for building EXISTS subqueries with Prisma's tagged templates.

Correct Implementation Pattern

timesheetMatrixView.ts — parameterized filter pattern
// User Group filter — correct parameterized approach
if (where.userGroupId?.length) {
  clauses.push(
    Prisma.sql`EXISTS (
      SELECT 1 FROM \`_User_UserGroup\`
      WHERE \`A\` = \`User\`.\`id\`
      AND \`B\` IN (${Prisma.join(where.userGroupId)})
    )`
  );
}

// Project Lead filter — reuses buildJobProjectExistsClause pattern
if (where.projectLeadId?.length) {
  clauses.push(
    Prisma.sql`EXISTS (
      SELECT 1 FROM \`Job\` j
      INNER JOIN \`Project\` p ON j.\`projectId\` = p.\`id\`
      WHERE j.\`id\` = tdr.\`jobId\`
      AND p.\`projectLeadId\` IN (${Prisma.join(where.projectLeadId)})
    )`
  );
}
Dev: Both filters support multi-select (array of IDs). Prisma.join() correctly handles parameterization of array values. Empty arrays should skip the clause entirely (guard with ?.length).
